# v0.4 要件定義書レビュー及びMVP向け改善案

## 1. 既存設計 (v0.4) の課題（個人開発・直列実行のコンテキストにおいて）

「個人開発」かつ「完全直列実行のみ」という前提を踏まえても、v0.4の設計には運用を不安定にする以下の脆弱性が残っています。

1. **状態推測による誤作動の危険性:** PRの有無やレビュー状態から「次にどのエージェントを起動するか」を推測する設計は、人間が手動でPRを触ったり、別の自動化ツールが干渉したりした際に容易に状態を見失います。
2. **脆い紐付けロジック:** ブランチ名の文字列パース（`<type>/<issue番号>-<概要>`）によるIssueとPRの紐付けは、タイポ一つで破綻する脆弱なアプローチです。
3. **文脈の喪失:** オーケストレーター（CLI）での対話内容がIssue本文に全て乗り切らない場合、後続のコーダーエージェントへ意図が正しく伝わりません。

※なお、直列実行前提であるため、高度なファイルI/Oの競合対策は不要です。

## 2. 改善案 (v0.4.1 提案): ローカル環境に最適化したラベル駆動MVP

GitHub ActionsやWebhooksの導入（当初のv0.5案）は、ローカル開発環境の利点（ローカルファイルの直接操作等）を損ない、インフラ構築のオーバーヘッドが大きすぎる「オーバーエンジニアリング」と判断しました。

MVPとして、v0.4の「ローカルでのポーリング」を維持しつつ、状態管理を堅牢にするアプローチを提案します。

### コアコンセプト

1. **明示的なラベルによる状態管理への回帰:** 推測を廃止し、明確なトリガーラベル（`needs-coder`, `needs-reviewer`など）をポーリングの検索条件（`gh issue list --label <label>`）として使用します。
2. **GitHub標準の紐付け機能の利用:** ブランチ名のパースを廃止し、PR作成時に `gh pr create --issue <Issue番号>` コマンドを明示的に使用させます。
3. **ローカルコンテキストファイルの活用:** Issueに書ききれない詳細な設計指示は、`.agents/context/<Issue番号>.md` のようなローカルファイルに書き出し、コーダーがそれを読み込むようにします。
4. **簡易ロック（Fail Fast）:** 二重起動防止のため、タスク実行中は単一の `worker.lock` ファイルを生成し、ロック存在時はポーリング処理をスキップ（Fail Fast）させます。

### 状態遷移（ラベル）の定義

状態遷移は「ラベルの付与」によってのみ行われます。ポーリングはこのラベルを探します。

| 現在のラベル | 意味・状態 | Daemonの振る舞い |
|---|---|---|
| `needs-coder` | オーケストレーター完了、または差戻しによる修正待ち。 | コーダーをローカルプロセスで起動。 |
| `needs-reviewer`| PR作成完了。レビュー待ち。 | レビュアーをローカルプロセスで起動。 |
| `approved` | レビュー承認済み。 | **何もしない。人間によるマージ判断を待つ。** |

### ワークフローの改善点

* **コーダーの終了処理:** 実装完了後、`gh pr create --issue <Issue番号>` でPRを作成し、対象Issueのラベルを `needs-coder` から `needs-reviewer` に**付け替えて**終了します。
* **レビュアーの終了処理:** レビュー結果が `APPROVED` ならIssueラベルを `approved` に、`CHANGES_REQUESTED` なら `needs-coder` に**付け替えて**終了します。
