# ADR-29455: Infer Minimal Secret Set for Call-Workflow Fan-Out Jobs

**Date**: 2026-05-01
**Status**: Draft
**Deciders**: Unknown (AI-generated from PR diff — verify before accepting)

---

## Part 1 — Narrative (Human-Friendly)

### Context

The `safe-outputs: call-workflow:` feature in gh-aw generates fan-out jobs that call reusable worker workflows. Previously, every generated orchestrator job used `secrets: inherit`, which forwards **all** secrets from the calling workflow to the worker — regardless of which secrets the worker actually uses. This violates the principle of least privilege: a worker that only needs `GH_AW_GITHUB_TOKEN` receives `CODEX_API_KEY`, `OPENAI_API_KEY`, and every other secret the orchestrator has access to. As the number of workers and secrets grows, the blast radius of any compromised worker or its dependencies increases correspondingly.

### Decision

We will change the compiler to infer the minimal set of required secrets for each `call-workflow` fan-out job instead of unconditionally emitting `secrets: inherit`. When compiling a worker workflow that has a `workflow_call` trigger, the compiler will perform a two-pass body generation: the first pass collects all `${{ secrets.* }}` references; the second pass injects `on.workflow_call.secrets` declarations (excluding `GITHUB_TOKEN`, which GitHub Actions provides automatically). When building orchestrator fan-out jobs, the compiler reads the worker's compiled lock file to extract these declarations and maps each secret explicitly (`NAME: ${{ secrets.NAME }}`). If the worker has no lock file or declares no secrets, the orchestrator falls back to `secrets: inherit` for backward compatibility.

### Alternatives Considered

#### Alternative 1: Keep `secrets: inherit` Universally

The simplest option is to leave the existing behaviour unchanged. `secrets: inherit` is fully supported by GitHub Actions and requires no additional compiler logic. This was rejected because it exposes every caller secret to every worker, violating least privilege and increasing the impact of any worker-side secret leak or supply-chain compromise.

#### Alternative 2: Require Authors to Declare Secrets Manually in Workflow Frontmatter

Workflow authors could explicitly list the secrets each worker needs inside the orchestrator markdown (e.g., a new `secrets:` field under `call-workflow:`). This keeps the compiler simple and puts authors in full control. It was not chosen because it places an ongoing maintenance burden on authors, is error-prone (secrets get added to workers but forgotten in orchestrators), and breaks the gh-aw promise that compilation is authoritative and self-contained.

#### Alternative 3: Static Analysis of Worker Markdown Source

The compiler could scan the worker's `.md` source for `${{ secrets.* }}` patterns at orchestrator-compile time without a separate worker compilation step. This was not chosen because it would require the compiler to re-implement secret collection logic outside the normal compilation pipeline, would miss secrets injected by the compiler itself (e.g., from imported shared workflows), and would create an unreliable dependency on source layout rather than the canonical compiled output.

### Consequences

#### Positive
- Workers receive only the secrets they declare, reducing secret exposure surface and blast radius.
- The mapping is self-maintaining: when a worker's secret usage changes, recompiling the worker updates its lock file, and the next orchestrator compilation picks up the correct set automatically.
- Backward compatible: orchestrators still emit `secrets: inherit` for workers that have not yet been compiled with the new feature, so no existing workflows break.
- Explicitness improves auditability — the orchestrator lock file now lists exactly which secrets flow to each worker job.

#### Negative
- Two-pass compilation adds runtime overhead and code complexity inside `generateYAML`.
- The orchestrator compilation now has an implicit dependency on the worker's compiled lock file being present and up-to-date. If a worker is modified but not recompiled before the orchestrator is recompiled, the stale secrets mapping silently falls back to `secrets: inherit`.
- `GITHUB_TOKEN` exclusion logic must be maintained separately from the rest of the secret-collection pipeline and is easy to misapply if new auto-provided tokens are added in the future.

#### Neutral
- The `on.workflow_call.secrets` declarations generated by this feature will appear in every reusable worker's lock file going forward, increasing lock-file size slightly.
- User-defined `on.workflow_call.secrets` entries in the source are preserved and merged with auto-generated ones, giving authors an escape hatch to set `required: true` for specific secrets.

## Part 2 — Normative Specification (RFC 2119)

> The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in this section are to be interpreted as described in [RFC 2119](https://www.rfc-editor.org/rfc/rfc2119).

### Worker Compilation — Secret Declaration Injection

1. When compiling a workflow that contains a `workflow_call` trigger, the compiler **MUST** collect all `${{ secrets.* }}` references from the generated workflow body and inject corresponding entries into the `on.workflow_call.secrets` section of the compiled output.
2. The compiler **MUST NOT** include `GITHUB_TOKEN` in the injected `on.workflow_call.secrets` declarations, as it is automatically provided by GitHub Actions and cannot be passed as an explicit caller secret.
3. Each injected secret entry **MUST** be declared with `required: false` unless the workflow author has explicitly set a different value in the source.
4. If the workflow author has declared secrets in `on.workflow_call.secrets` in the source, those declarations **MUST** be preserved and **MUST** take precedence over auto-generated entries for the same secret name.
5. The compiler **MUST** regenerate the workflow body after injecting `on.workflow_call.secrets` to ensure the compiled output reflects the updated `on:` section.

### Orchestrator Compilation — Explicit Secret Mapping

1. When building a `call-workflow` fan-out job, the compiler **MUST** attempt to read the worker's compiled lock file (`.lock.yml`) to extract its `on.workflow_call.secrets` declarations.
2. If the worker's lock file exists and declares one or more secrets, the orchestrator **MUST** map each declared secret explicitly using the form `NAME: ${{ secrets.NAME }}` and **MUST NOT** emit `secrets: inherit` for that job.
3. If the worker's lock file does not exist, cannot be read, or declares no secrets in `on.workflow_call.secrets`, the orchestrator **MUST** fall back to `secrets: inherit` for that job to maintain backward compatibility.
4. The compiler **MUST** log a warning when it falls back to `secrets: inherit` due to a read error, to aid debugging.
5. Orchestrators **MUST NOT** pass secrets that are not declared in the worker's `on.workflow_call.secrets` section when explicit mapping is in use.

### Conformance

An implementation is considered conformant with this ADR if it satisfies all **MUST** and **MUST NOT** requirements above. Failure to meet any **MUST** or **MUST NOT** requirement constitutes non-conformance.
