# ADR-38325: Require Operator-Authored Justification to Disable the Agent Sandbox

**Date**: 2026-06-10
**Status**: Draft

## Context

Disabling the coding agent sandbox via `sandbox.agent: false` removes the agent firewall — a security trust boundary that isolates the AI engine from the network. Previously two paths let a workflow remove this boundary without any reviewable record of operator intent: the `dangerously-disable-sandbox-agent` feature accepted a boolean `true` as a legacy shorthand, and the `network-firewall-migration` codemod silently injected a generic justification string (`"migrated from deprecated network.firewall disable setting"`) when migrating workflows from the deprecated `network.firewall: false` setting. In both cases the resulting frontmatter looked compliant but carried no human explanation of *why* the firewall was disabled, defeating the purpose of requiring a justification at all.

## Decision

We will require an operator-authored literal justification string to disable the agent sandbox. The `dangerously-disable-sandbox-agent` feature value **must** be a static string of at least 20 characters after trimming whitespace; boolean values (`true`/`false`) and `${{ expressions }}` are rejected at compile time. The validated justification is stored on the parsed `AgentSandboxConfig.DisableReason` struct field so it is available to diagnostics, logging, and audit rather than being discarded after the compilation gate. The compiler diagnostic explicitly states that disabling the sandbox *removes a trust boundary*. The migration codemod no longer invents a justification: `ensureSandboxDisableFeatureFlag` and the `migratedSandboxDisableJustification` constant are removed, so a workflow migrated from `network.firewall: false` fails compilation until the operator writes their own reason, while pre-existing operator-authored justifications are preserved.

## Alternatives Considered

### Alternative 1: Keep the boolean shorthand and auto-inject a migration justification
Continue accepting `dangerously-disable-sandbox-agent: true` and have the codemod insert a generic justification automatically. Rejected because it produces a justification with no real operator intent — the frontmatter passes review while communicating nothing about *why* the trust boundary was removed, which is the entire reason the justification field exists.

### Alternative 2: Allow expressions (e.g. `${{ inputs.reason }}`) as justifications
Permit dynamic GitHub Actions expressions so the reason could be supplied at runtime. Rejected because the justification must be a permanent, reviewable record visible at code-review and compile time; an expression cannot be audited from the source frontmatter and could resolve to an empty or meaningless value.

## Consequences

### Positive
- Removing the agent sandbox now leaves a permanent, reviewable record of operator intent in the workflow frontmatter.
- The justification is retained on `AgentSandboxConfig.DisableReason`, making it available to diagnostics and audit logging rather than being checked once and dropped.
- The compiler diagnostic frames the change as removing a security trust boundary, improving operator awareness at the point of failure.

### Negative
- Workflows migrating from the deprecated `network.firewall: false` setting now fail compilation until the operator manually writes a justification, introducing a breaking step in the migration path.
- The 20-character minimum is a heuristic for "meaningful" and does not prevent low-effort placeholder text that technically satisfies the length check.

### Neutral
- Pre-existing operator-authored justifications are preserved by the codemod, so already-migrated workflows are unaffected.
- The documentation (`docs/src/content/docs/reference/sandbox.md`) is updated to describe the required justification, the rejected value forms, and a warning about the trust-boundary implication.
