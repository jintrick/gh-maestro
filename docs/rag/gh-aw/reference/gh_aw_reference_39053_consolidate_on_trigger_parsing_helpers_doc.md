# ADR-39053: Consolidate Duplicated `on.*` Trigger Parsing Into Shared Helpers

**Date**: 2026-06-13
**Status**: Draft

## Context

The trigger-specific extractors in `pkg/workflow/frontmatter_extraction_yaml.go` (deployment status state, workflow-run conclusion, slash/legacy command, and label command) each independently re-implemented the same two pieces of plumbing: walking the nested `frontmatter["on"].<trigger>` map structure, and normalizing a field that may be either a single scalar string or an array of strings. Because this logic was copy-pasted across four call sites, the extractors were at risk of drifting apart — for example, one handling non-string array entries differently from another. This PR refactors that shared plumbing without changing observable compilation behavior.

## Decision

We will introduce three small, package-private helpers and reuse them across the trigger extractors:

- `extractOnTriggerValue(frontmatter, trigger)` — returns the raw `on.<trigger>` value when an `on` map is present.
- `extractOnTriggerMap(frontmatter, trigger)` — returns the `on.<trigger>` value as a `map[string]any` for object-syntax triggers.
- `normalizeStringOrStringSlice(raw)` — converts a scalar string or string-like array into `[]string`, ignoring non-string array elements.

Each trigger-specific extractor is narrowed to its own validation and expression/config rendering, delegating traversal and string normalization to these helpers. Existing behavior — including silently ignoring non-string entries in mixed arrays and the legacy `command:` deprecation warning — is preserved.

## Alternatives Considered

### Alternative 1: Leave the duplication in place
Keep each extractor self-contained with its own inline traversal and normalization. Rejected because the duplicated logic had already begun to diverge in small ways and was a maintenance hazard each time a new trigger field was added.

### Alternative 2: Decode `on.*` into typed structs via a YAML/`mapstructure` unmarshal
Replace ad-hoc `map[string]any` walking with strongly-typed structs decoded from the frontmatter. Rejected as too large a change for this refactor: the frontmatter is intentionally loosely typed (scalar-or-array fields, optional keys, legacy aliases), and struct decoding would require broader changes and risk altering edge-case behavior the current code deliberately tolerates.

## Consequences

### Positive
- Single source of truth for `on.<trigger>` traversal and scalar-or-array normalization, reducing the chance of behavioral drift between extractors.
- Adding or modifying a trigger extractor is now shorter and less error-prone.
- New focused unit tests cover the shared helpers and label-command extraction directly.

### Negative
- One extra layer of indirection between an extractor and the underlying map access, slightly reducing locality when reading a single extractor in isolation.
- The shared helpers now carry responsibility for all edge cases (non-string array entries, missing/`non-map` `on` sections); a bug there affects every trigger at once.

### Neutral
- No change to compiled workflow output or the public compiler API; this is an internal refactor of `pkg/workflow`.
- ADR is numbered by pull request number (`39053`) per repository convention.
