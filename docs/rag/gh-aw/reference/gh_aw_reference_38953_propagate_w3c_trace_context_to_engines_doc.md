# ADR-38953: Propagate W3C Trace Context into Engine Execution Steps

**Date**: 2026-06-12
**Status**: Draft

## Context

gh-aw already emits OpenTelemetry data: the setup action writes `GITHUB_AW_OTEL_TRACE_ID` and `GITHUB_AW_OTEL_PARENT_SPAN_ID` to `GITHUB_ENV`, and these values correctly parent the MCP gateway spans under the `gh-aw.agent.setup` span. However, this trace context was never propagated to the engine process itself (claude, codex, copilot, crush, gemini, opencode, antigravity, pi). As a result, engine-internal spans — for example Claude Code's `claude_code.interaction`, `claude_code.llm_request`, and `claude_code.tool` — were orphaned from the workflow trace, breaking the end-to-end distributed trace in the OTEL backend. The W3C Trace Context standard defines a `TRACEPARENT` environment variable that engines implementing the standard will honor as their parent span context.

## Decision

We will propagate the W3C trace context into every engine execution step by injecting a `TRACEPARENT` environment variable. A single shared helper, `applyTraceContextEnvToMap` in `safe_outputs_env.go`, sets `TRACEPARENT` to a GitHub Actions expression that resolves to the W3C traceparent format `00-<trace-id>-<span-id>-01` when both OTEL IDs are present, and to an empty string (a safe no-op) otherwise. The helper is called from all eight engines immediately after `applySafeOutputEnvToMap`, before user-supplied `engine.env` overrides are merged via `maps.Copy`, so that a user-provided `TRACEPARENT` naturally wins. This parents engine spans under the existing `gh-aw.agent.setup` span, keeping engine spans and MCP gateway spans as siblings under the same trace root.

## Alternatives Considered

### Alternative 1: Per-engine inline TRACEPARENT injection

Each engine could construct and set the `TRACEPARENT` value directly in its own `GetExecutionSteps` implementation. This was rejected because it duplicates the same conditional GitHub Actions expression across eight files, making the W3C format and fallback logic error-prone to keep consistent and harder to evolve. A single shared helper centralizes the format in one place.

### Alternative 2: Compute TRACEPARENT in Go at compile time

The traceparent string could be assembled in Go rather than as a runtime GitHub Actions expression. This was rejected because `GITHUB_AW_OTEL_TRACE_ID` and `GITHUB_AW_OTEL_PARENT_SPAN_ID` are only known at workflow run time (written to `GITHUB_ENV` by the setup action), not at compile time. The conditional expression defers evaluation to the runner and degrades gracefully to an empty string when OTEL is not configured.

## Consequences

### Positive
- Engine-internal spans now nest under `gh-aw.agent.setup`, producing a complete end-to-end distributed trace.
- The W3C traceparent format and its no-OTEL fallback live in a single helper, so all eight engines stay consistent.

### Negative
- A new `TRACEPARENT` line appears in every compiled engine step, requiring all affected golden snapshots to be regenerated and kept in sync.
- The behavior depends on each engine honoring W3C `TRACEPARENT`; engines that do not implement the standard silently gain no tracing benefit.

### Neutral
- The helper sets `TRACEPARENT` unconditionally; user `engine.env` overrides take effect only because they are merged afterward via `maps.Copy`, coupling correct precedence to call ordering.
- When OTEL is not configured, the expression resolves to an empty string, which compliant engines treat as "no parent context."
