# ADR-36699: Infer Copilot SDK Driver Runtime from Filename Extension

**Date**: 2026-06-03
**Status**: Draft

## Context

The `engine.copilot-sdk-driver` option let workflow authors supply a custom Copilot SDK driver script, but it accepted only Node.js files (`.js`, `.cjs`, `.mjs`) and always executed them via the resolved Node binary. Authors who prefer Python, TypeScript, Ruby, or who already have a purpose-built executable on `PATH` had no way to plug it in without writing a Node.js shim. The driver name is also security-sensitive — it is interpolated into a generated shell command — so any expansion of accepted values has to preserve the "safe basename, no path separators or shell metacharacters" guarantee. We needed a way to support more driver languages without introducing a new configuration surface or weakening that validation.

## Decision

We will infer the driver's runtime and its SDK install command from the driver filename's extension rather than adding an explicit runtime field. A relaxed schema pattern makes the extension optional, and Go validation enforces a strict allowlist: `.js/.cjs/.mjs` run under Node, `.py` under `python3`, `.ts/.mts` under `ts-node`, `.rb` under `ruby`, and a bare name (no extension) is treated as an arbitrary executable already in `PATH` (invoked directly, with no setup-action-directory prefix). The same extension drives `buildCopilotSDKInstallStep`, so a `.py` driver triggers `pip install` while everything else falls back to the Node.js `npm install` of the SDK.

## Alternatives Considered

### Alternative 1: Keep Node-only and require JS wrappers
Leave the driver Node-only and ask authors to shell out to other interpreters from a small `.cjs` wrapper. Rejected because it pushes boilerplate and a second process layer onto every non-JS author, and obscures the real driver behind a stub.

### Alternative 2: Add an explicit `runtime:` configuration field
Let authors declare the interpreter explicitly (e.g. `copilot-sdk-driver-runtime: python3`) instead of inferring it. Rejected because it adds a second config field that can contradict the filename, requires its own validation and documentation, and the extension already conveys the intent unambiguously for the supported languages.

## Consequences

### Positive
- Authors can write SDK drivers in Python, TypeScript, or Ruby, or point at any pre-installed executable, with no JS shim.
- No new configuration field — the existing single `copilot-sdk-driver` value carries the runtime choice via its extension.
- Validation stays strict: unknown extensions (`.sh`, `.java`, …) are rejected with a descriptive error, and the safe-basename checks are unchanged.

### Negative
- Ruby, TypeScript, and arbitrary-command drivers fall back to the Node.js `npm install` of the SDK; only Python gets a language-specific (`pip`) install, so other runtimes may need their SDK provisioned by other means.
- The selected interpreter (`python3`, `ts-node`, `ruby`) and any bare-command executable are assumed to be present on the runner; a missing runtime surfaces only at execution time.

### Neutral
- The driver name continues to be interpolated into a generated shell command; the extensionless "arbitrary command" path deliberately skips the setup-action-directory prefix and runtime wrapper.
- Behavior is covered by new unit tests for schema validation, engine validation, execution-step generation, and install-step selection across each supported extension.
