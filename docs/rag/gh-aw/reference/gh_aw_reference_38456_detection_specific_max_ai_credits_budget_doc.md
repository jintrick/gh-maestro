# ADR-38456: Detection-Specific max-ai-credits Budget with Runtime Override

**Date**: 2026-06-10
**Status**: Draft

## Context

The AWF API proxy enforces a per-run AI Credits budget (`apiProxy.maxAiCredits`) for agent runs, defaulting to `1000` credits and overridable enterprise-wide via the `GH_AW_DEFAULT_MAX_AI_CREDITS` GitHub Actions variable. Threat-detection runs (`safe-outputs.threat-detection`) previously shared this single budget control, so there was no way to cap detection spend independently from the main agent. Detection runs are typically lighter and more uniform than agent runs, so coupling them to the agent budget either over-provisions detection or forces operators to compromise the agent budget to constrain detection. This PR introduces a dedicated budget control so the two can be tuned independently.

## Decision

We will add a dedicated `safe-outputs.threat-detection.max-ai-credits` frontmatter field with its own built-in default of `400` credits and its own enterprise runtime override variable, `GH_AW_DEFAULT_DETECTION_MAX_AI_CREDITS`. The compiler resolves the detection budget with a precedence chain that mirrors the existing main-agent chain: an explicit frontmatter value wins at compile time; otherwise the compiler emits the runtime expression `${{ vars.GH_AW_DEFAULT_DETECTION_MAX_AI_CREDITS || '400' }}` so the GitHub Actions variable is resolved at run time. Main-agent `max-ai-credits` behavior is left unchanged.

## Alternatives Considered

### Alternative 1: Reuse the existing `max-ai-credits` for detection
Detection runs would continue to share the agent budget and its `GH_AW_DEFAULT_MAX_AI_CREDITS` override. This is the simplest option and adds no new configuration surface, but it makes it impossible to cap detection independently — operators cannot lower detection spend without also lowering the agent budget (or vice versa). Rejected because independent tunability is the core requirement.

### Alternative 2: Hardcode a detection default with no runtime override
Detection would use a fixed built-in default (e.g. `400`) with only frontmatter overriding it. This avoids the second `vars.*` variable, but enterprises could not set an organization- or repository-wide detection default without editing every workflow's frontmatter — the same gap the existing `GH_AW_DEFAULT_*` variables were created to close. Rejected for inconsistency with the established enterprise-defaults pattern.

## Consequences

### Positive
- Detection budget can be tuned independently of the agent budget, with a lower default (`400`) that reflects typical detection cost.
- Enterprises can set an organization- or repository-wide detection default via `GH_AW_DEFAULT_DETECTION_MAX_AI_CREDITS` without editing individual workflows, consistent with the existing `GH_AW_DEFAULT_*` family.

### Negative
- Adds a second, closely-named budget variable and precedence chain that must be documented and kept in sync with the main-agent path, increasing maintenance surface and the chance of operator confusion between the two.
- Introduces a divergent built-in default (`400` for detection vs `1000` for agents) that contributors must remember when reasoning about budgets.

### Neutral
- The resolution mechanism (compile-time literal → `vars.*` runtime expression → built-in constant) is identical in shape to the main-agent path, so the new code reuses the established expression-builder and injection helpers rather than introducing a new pattern.
- The CLI `gh aw env` defaults file gains a `default_detection_max_ai_credits` key, extending the existing batch-management surface.
