# ADR-36538: Derive Copilot SDK Permission Model in the Engine

**Date**: 2026-06-03
**Status**: Draft

## Context

Copilot SDK mode was effectively running with a broad "allow everything" permission posture despite workflow-level tool constraints, because the SDK driver wired an unconditional `onPermissionRequest: approveAll` handler. At the same time, the JavaScript harness was re-parsing Copilot CLI flags (`--allow-tool`, `--allow-all-tools`) from `serverArgs` to derive permissions, duplicating logic the Go engine already owns. This split responsibility risked drift between the permissions the engine intended and what the SDK session actually enforced, and gave no visibility into denied requests. We need SDK-mode permissions to faithfully reflect engine-generated tool configuration.

## Decision

We will derive a normalized permission model in the Go engine and pass it explicitly to the SDK driver. The engine emits a `permissionConfig` (`allowAllTools` plus a deduplicated, sorted `allowedTools` list) inside the SDK stdin JSON payload via `buildCopilotSDKPermissionConfig`, so the JavaScript side no longer parses CLI flags for permission derivation. The SDK driver replaces unconditional `approveAll` with a rule-aware handler that is only wired when explicit permission config exists, defaulting to deny for non-allowlisted shell, MCP, `web_fetch`, and write/custom-tool requests, and surfacing denials through GitHub Actions-style core logging.

## Alternatives Considered

### Alternative 1: Keep JS-side flag parsing
Continue deriving permissions in the harness by parsing `--allow-tool`/`--allow-all-tools` from `serverArgs`. Rejected because it duplicates engine logic in two languages, is prone to drift, and couples permission semantics to CLI flag string formats rather than a structured contract.

### Alternative 2: Keep unconditional `approveAll`
Retain the simple always-approve handler and rely solely on which tools the engine wires up. Rejected because it makes SDK mode silently broader than the configured allowlist, undermining workflow-level tool constraints and providing no denied-request observability.

## Consequences

### Positive
- Single source of truth: the Go engine owns permission derivation, eliminating cross-language drift.
- Default-deny posture in SDK mode aligns enforcement with explicit allowlists across shell, MCP, `web_fetch`, and write/custom tools.
- Denied-request diagnostics via `core.info`/`core.warning` improve debuggability.

### Negative
- Adds a new structured `permissionConfig` contract that the engine and SDK driver must keep in sync as tool types evolve.
- More moving parts in the SDK driver (rule-aware handler) increase surface area for permission-matching bugs.

### Neutral
- When no toolsets/permission rules are specified, SDK default behavior is preserved (no `onPermissionRequest` injected), matching command-line mode semantics.
- `allowAllTools` is still honored via `approveAll` when the engine requests full tool access.
