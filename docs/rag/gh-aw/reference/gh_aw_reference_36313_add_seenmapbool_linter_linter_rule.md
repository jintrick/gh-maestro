# ADR-36313: Add a dedicated `seenmapbool` analysis-pass linter

**Date**: 2026-06-01
**Status**: Draft

## Context

The codebase maintains a set of custom `golang.org/x/tools/go/analysis` linters under `pkg/linters/`, each implemented as its own package and registered centrally in `cmd/linters/main.go`. A recurring memory-allocation anti-pattern was observed: `map[string]bool` is frequently used purely as a membership set (every write assigns the literal `true`), where `map[string]struct{}` is the idiomatic, zero-value-per-entry alternative. Code scanning surfaced 18 occurrences across `pkg/` and `cmd/`, and discussion mining (issues #36022, #36127, #36160) independently flagged the hand-rolled deduplication pattern that relies on this idiom. There was no automated guard preventing the pattern from recurring.

## Decision

We will add a new standalone analysis-pass linter, `seenmapbool`, in its own package `pkg/linters/seenmapbool/`, and register it in the central analyzer list in `cmd/linters/main.go`. The analyzer walks each function body (`*ast.FuncDecl` and `*ast.FuncLit`), collects local `map[string]bool` declarations via type information (`pass.TypesInfo`), and reports a variable only when **every** indexed write assigns the boolean literal `true`. Maps that ever assign `false` (genuine tri-state bool maps) are excluded. Test files are skipped via `filecheck.IsTestFile`, and the package ships `analysistest`-based fixtures, consistent with sibling linters.

## Alternatives Considered

### Alternative 1: Rely on an off-the-shelf linter (e.g. golangci-lint / gocritic)
Third-party linter aggregators offer overlapping set-vs-bool heuristics. It was not chosen because the repository has deliberately standardized on a homegrown `pkg/linters/` analysis-pass framework, giving full control over rule semantics, diagnostic wording, and CI integration without taking on an external linter aggregator dependency. *[TODO: verify whether an off-the-shelf equivalent was explicitly evaluated and rejected.]*

### Alternative 2: Add the check to an existing linter package
The set-detection rule could have been folded into a broader existing analyzer rather than introducing a new package. It was not chosen because each rule in `pkg/linters/` is intentionally a single-responsibility, independently testable, and individually toggleable package, which keeps rules discoverable and consistent with the established one-package-per-linter convention.

## Consequences

### Positive
- Automated, repeatable prevention of the `map[string]bool`-as-set pattern, removing reliance on ad-hoc review.
- Memory savings: `map[string]struct{}` allocates no value byte per entry, which matters in hot deduplication loops.
- Consistent with the established one-package-per-linter convention, so the addition is low-friction for maintainers.

### Negative
- Detection is scoped to locally declared maps written via direct `m[k] = true` index assignments; maps passed as parameters, stored in structs, or written through aliases/closures are not tracked (false negatives).
- Value analysis only recognizes the bare identifier `true`; a `true` produced via a constant, function call, or boolean expression is treated as a non-set write, which can suppress a legitimate diagnostic. *[TODO: verify intended handling of constant/computed true values.]*
- Adds another custom linter to maintain in-house rather than leveraging an upstream-maintained equivalent.

### Neutral
- The existing 18 occurrences are not auto-fixed by this PR; they must be remediated separately once the linter is enforced in CI.
- The analyzer requires the `inspect` pass and type information, matching the resource profile of sibling linters.
- The rule deliberately does not flag `map[string]bool` whose values are sometimes `false`, preserving genuine bool-map use cases.
