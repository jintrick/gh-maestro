# ADR-38197: Enforce Four-Level AI Credit Resolution Order

**Date**: 2026-06-09
**Status**: Draft

## Context

The compiler supports two AI spending guardrails — `max-daily-ai-credits` and `max-ai-credits` (per-run) — but their resolution behavior was inconsistent: some paths read defaults at compile time via `os.Getenv`, some could be influenced by repository `aw.json`, and the built-in daily default did not match the expected operational value. This made it ambiguous which value would actually apply to a generated workflow, and prevented operators from overriding defaults at action runtime through GitHub Actions variables. A single, predictable precedence chain is required so that explicit workflow configuration always wins and operators retain a runtime override without recompiling.

## Decision

We will resolve both AI credit limits through a fixed four-level precedence order: workflow `frontmatter` → imported shared workflow configuration (first-wins across imports, resolved at compile time) → the `GH_AW_DEFAULT_MAX_*_AI_CREDITS` GitHub Actions variable (resolved at **action runtime**, not compile time) → a built-in constant. Neither limit will read from repository `aw.json`. The built-in defaults are set to `5000` for daily credits and `1000` for per-run credits. The runtime variable layer is emitted as a GitHub Actions `${{ vars.* || 'default' }}` expression rather than baked in at compile time, so the per-run path emits a local shell export (`GH_AW_MAX_AI_CREDITS="${{ vars.GH_AW_DEFAULT_MAX_AI_CREDITS || '1000' }}"`) and injects `${GH_AW_MAX_AI_CREDITS}` into the AWF config JSON when the field is absent from frontmatter and all imports.

## Alternatives Considered

### Alternative 1: Resolve defaults at compile time via `os.Getenv`
Read the `GH_AW_DEFAULT_*` values from the environment during compilation and bake the literal number into the generated lock file. Rejected because it freezes the default at compile time: operators could not change the credit budget by setting a repository/organization variable without recompiling and committing new lock files. It also couples the generated artifact to the machine that compiled it.

### Alternative 2: Allow `aw.json` to participate in resolution
Treat repository-level `aw.json` as one of the resolution sources for the two limits. Rejected to keep the precedence chain narrow and auditable — adding `aw.json` introduces a fourth configuration surface with unclear ordering relative to imports and runtime variables, increasing the risk of a value applying that neither the workflow author nor the operator expects.

## Consequences

### Positive
- Predictable, documented precedence: explicit frontmatter always wins, and the same chain governs both limits.
- Operators can change credit budgets at runtime by setting `GH_AW_DEFAULT_*` GitHub Actions variables without recompiling workflows.
- Generated lock files are no longer pinned to a compile-time environment value, improving reproducibility across machines.

### Negative
- The per-run path now emits an extra shell export line and string-interpolates `${GH_AW_MAX_AI_CREDITS}` into the AWF JSON, adding shell-escaping complexity (handled by `shellEscapeArgWithVarPreserved`) and a new failure surface in escaping logic.
- All 245 lock files plus golden fixtures had to be regenerated, producing a large diff and a one-time merge-conflict risk for in-flight branches.

### Neutral
- New `compilerenv` helpers (`BuildDefaultMaxDailyAICreditsExpression`, `BuildDefaultMaxAICreditsExpression`) and the `shellEscapeArgWithVarPreserved` shell helper formalize the expression/escaping contract.
- The behavior is now pinned by a normative specification (`ai-credits-specification.md` §9 and §10) with a conformance test matrix (T-AIC-DG-001–007, T-AIC-PR-001–007).
