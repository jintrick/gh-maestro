# ADR-38449: Prompt Copilot Users to Choose copilot-requests (Org Billing) vs PAT in the add-wizard

**Date**: 2026-06-10
**Status**: Draft

## Context

When a user selects the Copilot engine in the `gh aw add` interactive wizard, the wizard previously always prompted for a `COPILOT_GITHUB_TOKEN` Personal Access Token (PAT). However, organizations with Copilot seats can authenticate Copilot workflows through the GitHub Actions token by granting `permissions.copilot-requests: write` in the workflow frontmatter — no PAT is required in that case. Forcing every Copilot user through PAT setup added unnecessary friction and an extra secret to manage for org users who could rely on their existing Copilot billing seat. The wizard had no path to express "use org billing instead of a PAT."

## Decision

We will add an explicit authentication-method prompt to the add-wizard whenever the Copilot engine is selected. The user chooses between (a) **copilot-requests** — authenticate via the GitHub Actions token using the org's Copilot billing seat, or (b) **PAT** — store a fine-grained token as `COPILOT_GITHUB_TOKEN`. When the user picks copilot-requests, the wizard injects `permissions.copilot-requests: write` into the workflow frontmatter (idempotently, via `ensureCopilotRequestsWritePermission`) and skips the `COPILOT_GITHUB_TOKEN` secret-resolution step entirely. The choice is threaded end-to-end through `AddInteractiveConfig.UseCopilotRequests` → `AddOptions.AddCopilotRequestsPermission`.

## Alternatives Considered

### Alternative 1: Auto-detect org Copilot seat and skip the prompt
Automatically detect whether the org has Copilot seats available and silently choose copilot-requests when so. Rejected because seat availability cannot be reliably detected from the wizard context, and silently choosing an auth method (and silently mutating workflow permissions) is surprising; an explicit, documented choice is safer and more transparent.

### Alternative 2: Keep PAT-only (status quo)
Continue always requiring a PAT for Copilot. Rejected because it imposes unnecessary secret-management overhead on org users who already have a Copilot billing seat and could authenticate via the native Actions token.

## Consequences

### Positive
- Org users with Copilot seats can set up Copilot workflows with no PAT and no managed secret.
- Authentication uses the native GitHub Actions token rather than a long-lived personal token, reducing secret sprawl.
- The auth method is an explicit, recorded choice rather than implicit behavior.

### Negative
- Adds an extra prompt step to the Copilot path of the wizard.
- Introduces a second authentication code path that must be maintained alongside the PAT path.
- The permission-injection logic must robustly parse and rewrite workflow frontmatter and remain idempotent across re-runs.

### Neutral
- The PAT flow remains unchanged for users who select it.
- Permission injection is wired through new fields on `AddOptions` and `AddInteractiveConfig`, so callers outside the wizard default to the prior (no-injection) behavior.
