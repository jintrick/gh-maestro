# ADR-36421: Derive Effective Tokens from Raw Usage

**Date**: 2026-06-02
**Status**: Draft

## Context

Effective Tokens (ET) is a single normalized cost metric computed from raw token usage (input, cache read, cache write, output, reasoning) weighted by token-class weights and a per-model multiplier. Historically ET was sometimes treated as partially persisted data: the `agent_usage.json` fallback path trusted a precomputed `effective_tokens` field, and other paths consumed previously computed totals. Because the weights and multipliers used to derive ET can change over time (via merged multiplier files, environment overrides, or `aw_info` custom weights), any stored ET could silently drift away from what the *current* configuration would produce, yielding inconsistent and non-reproducible cost reporting.

## Decision

We will treat ET as **derived data**, not stored data. Whenever raw token-usage fields are present, ET is recomputed from those raw fields using the current token-class weights and per-model multipliers, and any stored `effective_tokens` value is ignored. Stored `effective_tokens` is consulted **only** as a fallback when no raw token fields exist. The canonical derivation subtracts cache reads from input once (`effective_input = max(input - cache_read, 0)`), applies the per-class weights, and multiplies by the per-model multiplier, falling back to a multiplier of `1.0` for unknown models (longest-prefix match otherwise). Multiplier configuration is loaded with a fixed precedence: merged multipliers file (`/tmp/gh-aw/model_multipliers.json`, overridable via `GH_AW_MERGED_MODEL_MULTIPLIERS_PATH`) → `GH_AW_MODEL_MULTIPLIERS` env var → built-in `model_multipliers.json`.

## Alternatives Considered

### Alternative 1: Trust stored ET, recompute only when absent
Continue to use the precomputed `effective_tokens` whenever it is present, recomputing only when the field is missing (the prior behavior). This was rejected because stored ET can be stale relative to the active weights/multipliers, producing reporting that silently disagrees with the current configuration and is not reproducible.

### Alternative 2: Recompute and migrate stored ET eagerly at write time
Recompute ET once when usage is written and persist the corrected value. Rejected because it does not solve drift for already-written records, requires a migration path, and still leaves a stored value that becomes stale the next time weights or multipliers change. Deriving on read keeps a single source of truth (raw usage + current config).

## Consequences

### Positive
- ET is always consistent with the current token-class weights and multipliers; no silent drift.
- Reporting becomes reproducible — the same raw usage plus the same config always yields the same ET.
- Multiplier source precedence is explicit and testable; reasoning tokens are now plumbed through and counted.

### Negative
- ET is recomputed on every read where raw usage exists, a small added cost versus reading a stored value.
- Previously reported totals change for records that carried a stale stored ET (e.g. a fixture total moved from 243846 to 237902), which may surprise consumers comparing against historical numbers.
- Correct ET now depends on raw token fields being present and accurate upstream; if raw fields are dropped, the system falls back to the stored value and the derived-data guarantee no longer holds.

### Neutral
- Reporting/output shape is unchanged; only the *source* of the ET value changed.
- The unknown-model fallback multiplier (`1.0`) and longest-prefix matching behavior are preserved.

> **Migration note:** This ADR references the legacy Effective Tokens (ET) terminology for historical context. gh-aw now uses AI Credits (AIC) as the primary cost metric.
