# ADR-36496: Centralize Effective-Token Limit Parsing With Shared K/M-Suffix Helpers

**Date**: 2026-06-02
**Status**: Draft

## Context

The `max-effective-tokens` and `max-daily-ai-credits` frontmatter fields each had their own parsing logic duplicated across three independent code paths: the JSON schema (`main_workflow_schema.json`), the Go compiler (`engine.go` and `daily_effective_workflow.go`), and the JavaScript runtime helper (`effective_token_limits.cjs`). The schema inlined the same `^[1-9][0-9]*([kKmM])?$` pattern twice, and the Go fields normalized short-form (`K`/`M`) suffixes inconsistently — one path expanded `100M` while another only accepted plain integers. This divergence meant a value accepted by schema validation could be silently ignored or normalized differently at runtime, producing confusing cost-enforcement behavior. The constraint is that all three layers must agree on what a valid limit is and on its canonical base-10 normalization, while preserving existing `-1` disable semantics and GitHub Actions expression support.

## Decision

We will establish a single source of truth for effective-token limit parsing in each layer and reuse it for both fields. In the JSON schema we introduce shared `$defs` (`positive_integer_with_km_suffix_or_expression`, `max_effective_tokens_limit`, `max_daily_effective_tokens_limit`) referenced by both fields. In Go we add `pkg/workflow/effective_token_limit_parser.go` with shared `normalizePositiveEffectiveTokenLimit` and `parseMaxEffectiveTokenLimitValue` helpers that both field parsers call. In JavaScript we consolidate parsing around a shared bigint-based `parsePositiveEffectiveTokenLimitBigInt`. All paths normalize short-form suffixes to canonical base-10 identically, so `100M` is accepted and expanded the same way everywhere. Reference docs are updated to present short-form notation (`5M`, `15M`, `25M`) as the default style.

## Alternatives Considered

### Alternative 1: Keep per-field inline parsing
Leave each field's parsing duplicated inline in the schema and in each Go/JS function. This was the status quo and required no new files, but it was the direct cause of the drift this PR fixes — every future change to accepted formats would need to be applied in four or more places, with no compiler or schema mechanism to keep them in sync. Rejected because it does not address the root cause.

### Alternative 2: Normalize only in the schema layer
Accept short-form only at validation time and have the schema (or a pre-processing step) rewrite values to plain integers before the Go/JS parsers see them. This would keep the runtime parsers simple, but the runtime still receives raw frontmatter and GitHub Actions expressions that resolve at execution time (after schema validation), so the runtime parsers must independently understand short-form regardless. Rejected because it cannot cover the expression and runtime-resolution cases.

## Consequences

### Positive
- Single source of truth per layer: changes to accepted formats are made once (one `$def`, one Go helper, one JS helper).
- Guaranteed consistency: schema validation, Go normalization, and JS normalization now agree on canonical base-10 output for the same input.
- Existing semantics preserved: `-1` disable, integer values, and `${{ ... }}` expressions continue to work unchanged.

### Negative
- One extra layer of indirection and a new Go file to navigate when reading the parsing flow.
- Go and JavaScript implementations remain two separate codebases that must be kept behaviorally in sync by hand; only tests (not the type system) enforce parity.

### Neutral
- Documentation now defaults to short-form notation, so existing long-form examples in user workflows differ stylistically from the docs (both remain valid).
- Schema default values remain expressed as plain integers (`25000000`) even though docs prefer `25M`.

> **Migration note:** This ADR references the legacy Effective Tokens (ET) terminology for historical context. gh-aw now uses AI Credits (AIC) as the primary cost metric.
