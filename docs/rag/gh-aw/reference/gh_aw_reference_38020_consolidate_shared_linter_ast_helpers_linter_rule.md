# ADR-38020: Consolidate shared linter AST helpers into `pkg/linters/internal/astutil`

**Date**: 2026-06-09
**Status**: Draft

## Context

The custom analysis-pass linters under `pkg/linters/` each grew their own copies of small AST/type helper functions â€” `isFmtErrorf`, `rhsExprForIndex`, `isStringLiteral`, and `isLocalObject`. Because the copies were edited independently over time, their semantics drifted: `errormessage` matched `fmt.Errorf` by identifier name only (so it misfired on import aliases and was fooled by locally shadowed `fmt` values), while `fmterrorfnoverbs` had already moved to a stricter `types.PkgName`-based resolution. Similarly, `lenstringzero`'s `rhsExprForIndex` lacked the single-RHS/index-0 mapping case that `tolowerequalfold`'s copy handled. This duplication meant a correctness fix in one analyzer silently left the others wrong, and the drift was already producing divergent behavior in `rhsExprForIndex` and `isFmtErrorf`.

## Decision

We will extract the duplicated helpers into a single shared package, `pkg/linters/internal/astutil`, exporting `IsLocalObject`, `RhsExprForIndex`, `IsStringLiteral`, and `IsFmtErrorf`, and update every affected analyzer (`tolowerequalfold`, `lenstringzero`, `errstringmatch`, `fmterrorfnoverbs`, `errormessage`) to consume them while deleting the local copies. Where copies disagreed, we standardize on the **stricter, more correct** implementation: `IsFmtErrorf` resolves `fmt` through `types.PkgName` so it honors import aliases and rejects shadowed locals, and `RhsExprForIndex` preserves the single-RHS + index-0 mapping case. This gives one authoritative implementation per helper and eliminates the drift class entirely.

## Alternatives Considered

### Alternative 1: Keep per-analyzer copies and sync manually
Leave each helper local to its analyzer and rely on review discipline to keep them aligned. Rejected because this is exactly the process that already failed â€” the copies drifted into materially different (and in `errormessage`'s case, incorrect) behavior, and nothing structurally prevents the next divergence.

### Alternative 2: Build the helpers on a third-party / `golang.org/x/tools` utility layer
Reuse or wrap an existing analysis utility library instead of writing a local shared package. Rejected because the helpers encode project-specific, opinionated semantics (the single-RHS mapping rule, the alias-aware `fmt.Errorf` check) that no general-purpose library provides, and an `internal/` package keeps the surface private and tailored without adding a dependency.

## Consequences

### Positive
- Single source of truth: a correctness fix to any helper now benefits every analyzer at once, removing the drift failure mode.
- Standardizes on the stricter semantics, fixing the `errormessage` false positives/negatives around aliased and shadowed `fmt.Errorf`.
- New `astutil_test.go` unit tests plus aliased/shadowed fixtures in `fmterrorfnoverbs` and `errormessage` testdata lock the edge-case behavior against regression.

### Negative
- All five analyzers now share a dependency on `pkg/linters/internal/astutil`; a change to a helper's contract can affect multiple linters simultaneously, so it requires broader regression awareness.
- Adds a new internal package and exported API surface to maintain, versus inline private helpers.
- Standardizing on the stricter `IsFmtErrorf` is a subtle behavior change for analyzers that previously used name-only matching; any code that depended (even accidentally) on the looser behavior would now be treated differently.

### Neutral
- The change is a pure refactor of detection plumbing; no new diagnostics or linter rules are introduced.
- Helpers remain syntactic/type-checker based, consistent with the existing single-`inspect`-pass design of `pkg/linters/`.
- Package placement under `internal/` keeps the helpers unexported outside the linters tree.
