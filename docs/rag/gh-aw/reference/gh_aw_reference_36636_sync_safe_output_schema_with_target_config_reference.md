# ADR-36636: Keep the Safe-Output JSON Schema in Sync with `SafeOutputTargetConfig`

**Date**: 2026-06-03
**Status**: Draft

## Context

Safe output types in gh-aw are validated at compile time against `pkg/parser/schemas/main_workflow_schema.json`, while their runtime behavior is implemented in Go (`SafeOutputTargetConfig` / `ParseTargetConfig`) and the CJS runtime (`repo_helpers.cjs`). These two surfaces drifted: the Go parser and the CJS runtime already accepted `target`, `target-repo`, and `allowed-repos` for cross-repository operations, but the JSON schema omitted them for several safe outputs (e.g. `resolve-pull-request-review-thread`, `assign-milestone`, `hide-comment`, `link-sub-issue`, and the `allowed-repos` field on a further eight types). Because the schema declares `additionalProperties: false`, valid configurations such as `target: "*"` or `target-repo:` were rejected with `Unknown properties` errors before they could ever be compiled. The schema — not the runtime — was the bottleneck.

## Decision

We will treat the JSON schema as a faithful mirror of the capabilities already implemented in `SafeOutputTargetConfig`, and add the missing `target`, `target-repo`, and `allowed-repos` properties to every safe output type that embeds it. The schema must accept exactly the configurations the runtime supports — no more (so `additionalProperties: false` is preserved) and no less (so valid cross-repo configs compile). Regression tests assert this contract across all 25 affected safe output types.

## Alternatives Considered

### Alternative 1: Relax `additionalProperties: false`
We could drop the strict-additional-properties constraint so the schema tolerates any extra fields, sidestepping the need to enumerate each property. This was rejected because it would silently accept genuinely invalid or misspelled keys (e.g. `taget-repo`), removing the early, actionable validation errors that the strict schema provides. The new `TestMainWorkflowSchema_SafeOutputsRejectsUnknownProperties` test deliberately locks in the strict behavior.

### Alternative 2: Generate the schema from the Go struct
We could auto-generate the safe-output portion of the schema directly from `SafeOutputTargetConfig`, eliminating drift structurally. This is the more durable fix, but it is a substantially larger change touching the schema-generation pipeline and was out of scope for closing the immediate validation gap. It remains a reasonable future ADR.

## Consequences

### Positive
- Valid cross-repository safe-output configurations (`target: "*"`, `target-repo:`, `allowed-repos:`) now compile instead of failing with `Unknown properties`.
- The schema, Go parser, and CJS runtime now agree on the accepted property set, reducing confusion for workflow authors.
- Regression tests pin the contract for all 25 embedding types, so a future omission is caught immediately.

### Negative
- The schema and Go struct are still kept in sync manually, so the same class of drift can recur for newly added properties or types.
- The schema file grows more verbose, with the `allowed-repos` definition repeated per safe output type rather than referenced from a shared `$def`.

### Neutral
- No runtime behavior changes; only the compile-time validation surface was widened to match existing capabilities.
- The fix is purely additive to the schema (plus tests), so existing workflows are unaffected.
