# ADR-36064: Decompose Oversized Linter Functions and Use Safe Type Assertions

**Date**: 2026-05-31
**Status**: Draft

## Context

The repository enforces a custom `largefunc` linter that caps functions at 60 lines. Fourteen `run` functions and their inline closures across `pkg/linters` and `pkg/parser` exceeded this limit. Separately, two analyzers cast `pass.ResultOf[inspect.Analyzer]` with a single-value type assertion that would panic on a type mismatch, and four `json.Marshal` calls in `pkg/parser/frontmatter_merge_test.go` ignored their error returns. These analyzers are static-analysis passes whose structure (an `insp.Preorder` callback that walks the AST) naturally accretes nested closures, which is what pushed them over the line limit.

## Decision

We will decompose each oversized analyzer `run` function by extracting its inline closures into named, package-level helper functions, following a consistent two-level delegation pattern: the `insp.Preorder` callback delegates to a named `inspectXFuncDecl`, which in turn delegates its `ast.Inspect` walk to a named `inspectXNode`. We will also replace single-value type assertions on analyzer results with the two-value (`v, ok :=`) form that returns a descriptive error instead of panicking, and check the previously ignored `json.Marshal` errors in tests with `t.Fatalf`. The primary driver is conformance with the existing `largefunc` linter and elimination of panic/error-swallowing paths.

## Alternatives Considered

### Alternative 1: Raise or disable the `largefunc` threshold for these files
We could exempt the analyzers from the 60-line limit (via nolint directives or a higher threshold) since AST-walking code is inherently nested. Rejected because it weakens a repo-wide quality gate and the nesting can be expressed just as clearly with named helpers, which also become independently testable.

### Alternative 2: Keep inline closures but shorten them in place
We could trim each `run` function below 60 lines without extracting named functions, relying on anonymous closures. Rejected because anonymous closures remain hard to test in isolation and the line count is dominated by the walk logic itself, which cannot be meaningfully shortened without extraction.

## Consequences

### Positive
- Each analyzer's logic is split into small, individually testable named functions, improving readability.
- Removing single-value type assertions eliminates a latent panic path, replacing it with a descriptive error return.
- Checked `json.Marshal` errors make test failures explicit rather than silently passing.

### Negative
- More top-level functions per package increase the symbol surface and add a layer of indirection to follow when reading a single analyzer end to end.
- The consistent `inspectXFuncDecl`/`inspectXNode` naming pattern must be maintained by future analyzer authors to keep the codebase uniform.

### Neutral
- No change to analyzer behavior or diagnostics; this is a structural refactor within the existing static-analysis framework.
- The two-value assertion error paths are effectively unreachable in normal operation (the analyzer dependency guarantees the type), but are now defensive rather than panicking.
