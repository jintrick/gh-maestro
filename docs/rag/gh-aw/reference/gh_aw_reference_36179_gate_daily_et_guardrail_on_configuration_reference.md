# ADR-36179: Gate the Daily Effective-Token Guardrail on Explicit Configuration

**Date**: 2026-06-01
**Status**: Draft

## Context

The daily effective-token (ET) guardrail step was emitted into the activation job of every compiled workflow, and the setup step always installed the `@actions/artifact` client whenever the guardrail code path was present. This meant workflows that never configured a `max-daily-ai-credits` threshold still paid for a no-op guardrail step and an unnecessary artifact-client install on every run. Enablement was previously detected only from the frontmatter-derived `MaxDailyEffectiveTokens` value, so a threshold supplied purely through a workflow `env:` declaration (`GH_AW_MAX_DAILY_AI_CREDITS`) was not recognized. We needed the guardrail to run only when it is actually configured, by either configuration channel, without incurring activation overhead otherwise.

## Decision

We will gate the daily ET guardrail step and the `safe-output-artifact-client` setup input on a compile-time-emitted GitHub Actions conditional expression, `${{ env.GH_AW_MAX_DAILY_AI_CREDITS != '' }}`, centralized as the `maxDailyEffectiveTokensConfiguredIfExpr` constant. We will also extend guardrail-enablement detection (`hasMaxDailyEffectiveTokensGuardrail`) to recognize explicit `env:` declarations of `GH_AW_MAX_DAILY_AI_CREDITS`, using line-based key matching (including quoted keys) that ignores comments and substring false positives. The conditional is threaded through a new `generateSetupStepWithArtifactClientCondition` helper while the existing `generateSetupStep` signature is preserved as a thin wrapper.

## Alternatives Considered

### Alternative 1: Always emit the step and early-exit at runtime in the script
Keep emitting the guardrail step unconditionally and let the `check_daily_effective_workflow_guardrail.cjs` script return early when the threshold is unset. This was rejected because the setup job would still install the artifact client and the activation job would still spin up a no-op step, defeating the cost-avoidance goal.

### Alternative 2: Decide enablement entirely at compile time and omit the step
Determine enablement solely from frontmatter at compile time and omit the step (and artifact-client input) from the lockfile when no threshold is set. This was rejected because it cannot react to a threshold supplied only via a workflow `env:` value at runtime; the compile-time `if` expression preserves that flexibility while still skipping execution.

## Consequences

### Positive
- Workflows without a configured threshold skip the guardrail step and avoid the `@actions/artifact` install cost.
- Thresholds configured via an explicit `env:` declaration are now recognized, broadening valid configuration channels.

### Negative
- Introduces a second setup-step code path (`generateSetupStepWithArtifactClientCondition`) and a duplicated conditional-expression constant to keep in sync.
- Env-config detection is heuristic string/line matching rather than structured YAML parsing, which can be fragile if env formatting diverges.

### Neutral
- Lockfile expectations change: the guardrail step now renders an `if:` line and the setup input renders the conditional expression instead of a literal `'true'`.
- Test coverage was added for env-configured guardrail workflows and for env-key detection behavior.
