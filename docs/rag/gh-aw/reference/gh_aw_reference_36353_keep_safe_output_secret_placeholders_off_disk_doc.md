# ADR-36353: Keep Safe-Output Secret Placeholders Out of Runtime config.json

**Date**: 2026-06-01
**Status**: Draft

## Context

When compiling safe-outputs, the generator wrote `${RUNNER_TEMP}/gh-aw/safeoutputs/config.json` using an *unquoted* heredoc whenever the config contained secret, GitHub-context, or workflow-input expressions. The shell expanded those `${VAR}` references at runtime, materializing the actual secret values (e.g. `WRITE_PROJECT_PAT`) as plaintext on disk inside the runner. Any later process, log, or artifact step that read the file could observe the resolved secret. The runtime already had the same environment variables available, so on-disk expansion was unnecessary as well as exposing. This ADR records the decision to stop persisting resolved secret material to disk.

## Decision

We will keep secret/input/context expressions as literal `${VAR}` placeholders in the on-disk `config.json` and resolve them only in memory when the safe-outputs runtime loads the file. Concretely: the Go compiler (`pkg/workflow/mcp_setup_generator.go`) always emits the config with a single-quoted heredoc so the shell never expands placeholders, passes the corresponding env vars through to the safe-outputs server step, and the JS loaders (`safe_outputs_config.cjs`) substitute `${VAR}` from `process.env` after parsing. Token-bearing fields (keys matching `/token/i`) are additionally redacted from debug logs so the exposure is not merely relocated from disk into logs.

## Alternatives Considered

### Alternative 1: Keep expanding values via an unquoted heredoc (status quo)
The previous behavior expanded placeholders at config-write time. It was simplest and required no runtime resolution, but it wrote plaintext secrets to disk — the exact exposure this change exists to eliminate. Rejected on security grounds.

### Alternative 2: Drop the config file and pass the entire config through an environment variable
The runtime could read the full JSON from an env var instead of a file, avoiding any on-disk material. This was rejected because the file-based config path supports existing dynamic behavior and tooling, env-var size/escaping limits are awkward for large configs, and the in-memory placeholder resolution achieves the security goal with a smaller, more localized change.

## Consequences

### Positive
- Secret values are no longer written to disk; only `${VAR}` placeholders are persisted.
- Token-bearing config fields are redacted from debug logs, preventing log-based re-exposure.
- Existing dynamic config behavior (inputs, context vars) is preserved.

### Negative
- Placeholder resolution now lives in the runtime loaders, adding logic and a dependency on the correct env vars being exported into every step that consumes the config.
- The compiler must emit the same env-var block in more than one place (config-write step and server step), increasing the surface for drift if the two diverge.
- A legitimate config string that literally contains `${SOMETHING}` could be unintentionally substituted by the in-memory resolver.

### Neutral
- Regression tests now assert the heredoc stays single-quoted and that placeholders remain literal on disk while resolving at runtime, locking in the new contract.
