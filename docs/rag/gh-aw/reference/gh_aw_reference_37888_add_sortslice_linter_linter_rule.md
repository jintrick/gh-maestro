# ADR-37888: Add a dedicated `sortslice` analysis-pass linter

**Date**: 2026-06-08
**Status**: Draft

## Context

The codebase maintains a set of custom `golang.org/x/tools/go/analysis` linters under `pkg/linters/`, each implemented as its own single-responsibility package and registered centrally in `cmd/linters/main.go`. A recurring non-idiomatic pattern was observed: calls to `sort.Slice` and `sort.SliceStable`, whose `func(i, j int) bool` callback receives raw integer indices with no compile-time guarantee about their validity relative to the slice element type. Since Go 1.21 the standard library `slices` package offers the type-safe `slices.SortFunc` / `slices.SortStableFunc`, whose `func(a, b E) int` callback receives the actual typed elements and follows the `cmp.Compare` sign convention. This module targets Go 1.26.3, well above the 1.21 floor, yet code-pattern scanning surfaced 47 `sort.Slice` and 6 `sort.SliceStable` calls across non-test source under `pkg/` and `cmd/`.

## Decision

We will add a new standalone analysis-pass linter, `sortslice`, in its own package `pkg/linters/sortslice/`, and register it in the central analyzer list in `cmd/linters/main.go`. The analyzer inspects every `*ast.CallExpr`, matches selector expressions of the form `sort.Slice` / `sort.SliceStable` (package identifier `sort` plus the method name), and reports a diagnostic recommending `slices.SortFunc` / `slices.SortStableFunc` respectively. Test files are skipped via `filecheck.IsTestFile`, `// nolint` directives are honored via the shared `nolint` index, and the package ships `analysistest`-based fixtures with `// want` annotations, consistent with sibling linters.

## Alternatives Considered

### Alternative 1: Enable an upstream linter (e.g. `gocritic` / `staticcheck`)
Off-the-shelf analyzers can flag legacy `sort` usage. This was not chosen because the repository's custom-linter convention favors single-responsibility, individually toggleable in-house passes, and enabling a broad upstream linter would activate a large set of unrelated rules rather than this one targeted check. *[TODO: verify whether an upstream rule covering `sort.Slice` was evaluated and rejected.]*

### Alternative 2: Fold the check into an existing linter package
The `sort.Slice` check could have been added to a broader existing analyzer rather than introducing a new package. It was not chosen because each rule in `pkg/linters/` is intentionally a single-responsibility, independently testable, and individually toggleable package, which keeps rules discoverable and consistent with the established one-package-per-linter convention.

### Alternative 3: Type-check the selector against the actual `sort` package
The current implementation matches on the syntactic identifier `sort` rather than resolving it through `pass.TypesInfo` to the standard-library `sort` package. A fully type-aware match was not chosen for this initial pass to keep the analyzer minimal, accepting that a locally-defined `sort` identifier could produce a false positive.

## Consequences

### Positive
- Automated, repeatable prevention of new `sort.Slice` / `sort.SliceStable` usage, steering contributors toward the type-safe `slices.SortFunc` / `slices.SortStableFunc`.
- Consistent with the established one-package-per-linter convention, so the addition is low-friction for maintainers.
- Reuses the shared `filecheck` and `nolint` helpers, so test-file skipping and suppression behave identically to sibling linters.

### Negative
- Adds another custom linter to maintain in-house rather than relying on an upstream-maintained equivalent.
- Detection is purely syntactic (identifier-based): a `sort` identifier that is not the standard-library package would be falsely flagged, and aliased imports of `sort` would be missed (false negatives).
- The existing 47 + 6 occurrences are not auto-fixed by this PR; they must be remediated separately once the linter is enforced in CI.

### Neutral
- The analyzer requires the `inspect` pass, matching the resource profile of sibling linters.
- The diagnostic is report-only (no suggested-fix/autofix is emitted in this PR).
- The rule does not flag the legitimate fixed-key helpers `sort.Strings` / `sort.Ints` / `sort.Float64s`, which have no type-safety gap.
