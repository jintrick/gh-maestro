# ADR-37368: Add `jobs.<job>.setup-steps` Injected Before Framework App Token Minting

**Date**: 2026-06-06
**Status**: Draft

## Context

When agentic workflows authenticate with a GitHub App, the framework mints an App token (`actions/create-github-app-token`) inside framework-generated jobs such as `agent`, `safe_outputs`, and `conclusion`. Some users need to run their own bootstrap logic — OIDC login, dynamic secret fetching, credential preparation — *before* that token is minted, but until now only the agent job had a practical pre-token injection hook. The existing `jobs.<job>.pre-steps` hook did not guarantee placement ahead of token minting for built-in jobs, and there was no consistent, explicit way to express "run these steps first" across both built-in and custom jobs.

## Decision

We will add a `jobs.<job-name>.setup-steps` hook that uses the same step schema as `steps`/`pre-steps`, and inject these steps immediately after the compiler-generated setup action when present, otherwise before the earliest `create-github-app-token` or `checkout` boundary. `setup-steps` is the preferred explicit shape; existing `pre-steps` continues to work as a distinct later insertion point. The safe-output job builder is reordered so configured setup/pre steps run before App token minting, and import-merge logic merges conflicting `setup-steps` and `pre-steps` arrays independently (imported first, then main, per field).

## Alternatives Considered

### Alternative 1: Reuse only the existing `pre-steps` hook
We could have relied solely on `pre-steps`. It was rejected because `pre-steps` semantics did not guarantee insertion before App token minting in built-in jobs, and overloading it would have changed existing behavior in a backward-incompatible and less explicit way.

### Alternative 2: Hardcode OIDC/credential support into the framework
We could bake specific OIDC/secret-fetch behavior directly into the framework job builders. This was rejected as too narrow and inflexible — a generic, user-controlled step hook supports OIDC, secret fetching, and arbitrary bootstrap logic without the framework needing to anticipate every credential pattern.

## Consequences

### Positive
- Built-in jobs (`agent`, `safe_outputs`, `conclusion`, etc.) can now run OIDC/secret-fetch/bootstrap steps before framework App token minting.
- Backward compatible — existing `pre-steps` users are unaffected; `setup-steps` adds an earlier, explicit hook without collapsing the two fields together.

### Negative
- Two closely related concepts (`setup-steps` and `pre-steps`) now coexist, which can confuse authors about which to use.
- Step-insertion ordering logic gains complexity (token-mint boundary detection, backward walk to list-item boundary, checkout-vs-token-mint precedence).

### Neutral
- Schema extended to validate `jobs.<job>.setup-steps` with the same step schema as `steps`.
- Permission inference and script scanning now also cover `setup-steps` alongside `pre-steps` for built-in jobs.
